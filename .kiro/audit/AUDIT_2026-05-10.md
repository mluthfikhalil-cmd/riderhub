# RiderHub тАФ Full Source Audit

**Date:** 2026-05-10
**Scope:** `src/` + root config. 22 source files reviewed.
**Context:** local source is an older pre-3D in-progress state. Live production runs a 3D build whose source is not on disk. This audit catalogues bugs/incomplete logic **in the local source only** тАФ the live bundle is pinned and untouched.

Severity legend:
- ЁЯФ┤ **Critical** тАФ crashes, data loss risk, auth/security issue, or TS compile error blocking build
- ЁЯЯа **High** тАФ feature broken / visibly degraded for user
- ЁЯЯб **Medium** тАФ logic flaw that still renders, silent failures, UX papercuts
- ЁЯЯв **Low** тАФ code smell, dead code, minor type issue

## A. Project-wide / infrastructure

### A1. ЁЯФ┤ `@expo/vector-icons` not declared in `package.json`
`package.json` lists no `@expo/vector-icons` dependency, yet 19 files import from it. Expo SDK 54 bundles this via the `expo` meta-package, so runtime works, but `tsc --noEmit` fails with `TS2307` for every file importing it. Also blocks any pure-`tsc` CI check.
**Fix:** add `"@expo/vector-icons": "^14.x"` to dependencies, or add ambient module declaration.

### A2. ЁЯФ┤ `.env.example` commits a real live Supabase anon key
```
EXPO_PUBLIC_SUPABASE_URL=https://wqnpyzjixjkjygeulfvo.supabase.co
EXPO_PUBLIC_SUPABASE_ANON_KEY=eyJhbGciOi...
```
Anon keys are not "secret" in the Supabase sense, but RLS posture is the only thing between this key and your data. The file is tracked in git (`.gitignore` excludes `.env.local` but not `.env.example`). Rotate when RLS policies are reviewed.

### A3. ЁЯЯа `tsconfig` is strict and 20+ files fail compile
`tsconfig` extends `expo/tsconfig.base` with `"strict": true`, but `tsc --noEmit` returns ~25 errors. Build still works because Metro / esbuild transpile without type-checking, but there is no typecheck gate. See B1/B2/B3/C2 for the specific errors.

### A4. ЁЯЯб `app.json` has `"output": "single"` for web
Expo Router's `"single"` output generates one HTML and client-side routes. Combined with `vercel.json` rewrite `"/(.*)" тЖТ "/index.html"`, this works but causes a cold-load of the full bundle on every deep link. No preload for subroutes. Not a bug, but worth noting before SEO/perf work.

### A5. ЁЯЯб Build cache-buster `BUILD_VERSION` string is stale
`App.tsx` line 25: `BUILD_VERSION = 'v4.0.20260428'`. This is the string that matched the non-3D build that overwrote production. It is neither auto-generated nor git-tied. Consider tying it to `git rev-parse --short HEAD` at build time, or at least bumping it on every push.

### A6. ЁЯЯв `App.tsx.bak`, `src/theme_backup/` exist on disk
Accumulated backup files committed to repo. Safe to delete once recovery branch lands, but keep for now.

## B. Root app wiring (`App.tsx`, navigation)

### B1. ЁЯФ┤ `linking` prefixes readonly тЖТ `NavigationContainer` type mismatch
`App.tsx:170` passes a deeply-readonly `linking` object (all array literals and `as const`) to `NavigationContainer.linking`, which requires `string[]` (mutable). TS2322 error. Runtime is fine but the type is wrong.
**Fix:** remove the `as const`, or cast `as LinkingOptions<RootStackParamList>`.

### B2. ЁЯФ┤ `RideReplayScreen` route typing breaks stack screen type inference
`App.tsx:191`: `<Stack.Screen name="RideReplay" component={RideReplayScreen} />` fails because `RideReplayScreen` declares `Props = { route: { params: { ride: any } }; navigation: any; }` directly instead of using `NativeStackScreenProps<RootStackParamList, 'RideReplay'>`. TS2322.
**Fix:** change `RootStackParamList.RideReplay` from `undefined` to `{ ride: Ride }`, then type the screen via `NativeStackScreenProps`.

### B3. ЁЯЯа `Landing` route isn't in `linking.config.screens`
`App.tsx:45-66`: `Login`, `Register`, `Main`, etc. are all mapped to paths, but `Landing` is missing. Deep-link to the landing page falls through to the 404-ish default. Minor because `Landing` is the default screen for unauthenticated users.

### B4. ЁЯЯб Tab icon on Home reroutes to `Profile` stack screen, not `Profile` tab
`HomeScreen.tsx:82`: `navigation.navigate('Profile')` from inside `MainTabs` works because Profile is a tab, but in several other places (like `navigation.navigate('Leaderboard')`) we navigate to stack screens from tab context. React Navigation 7 supports this via root navigator fallback, so it works at runtime, but there's no type-safety тАФ all `navigation` props are implicitly `any` or missing `navigate` generics.

### B5. ЁЯЯб `forceReady` pattern masks real auth failures
`App.tsx:148-152`: after 3s timeout, `forceReady = true` and the screen renders even if Supabase auth is still loading. Good for a stuck session, but hides bugs where `initAuth()` rejects silently. Recommend logging at timeout.

### B6. ЁЯЯб `needsOnboarding` check uses only `user_metadata.onboarded`
`App.tsx:164`: `const needsOnboarding = user && !user.user_metadata?.onboarded;`. `OnboardingScreen.handleSkip` updates this flag, but there's no re-check against `profiles.onboarded` (which is updated separately in `handleSave`). If metadata sync fails, user is stuck in a loop.

## C. Auth (`AuthContext.tsx`, `AuthScreens.tsx`, `LandingScreen.tsx`)

### C1. ЁЯФ┤ `signOut` calls `localStorage.clear()`
`AuthContext.tsx:126`: `localStorage.clear()` wipes **all** local storage, not just Supabase keys. This deletes `riderhub_cart_items`, `riderhub_liked_posts`, `riderhub_joined_<id>`, and anything else other features write. User loses state on every logout.
**Fix:** iterate keys and remove only those starting with `sb-` or containing `supabase` / `riderhub-auth`.

### C2. ЁЯЯб `signUp` never asks for email confirmation on native
`signUp` returns `needsVerification: !data.session`. Only `LandingScreen.doReg` shows the alert, and only on web. On mobile there's no feedback тАФ user sees the modal close and nothing else happens.

### C3. ЁЯЯб `AuthScreens.tsx` and `LandingScreen.tsx` both implement auth UI
Two different login/register flows live side-by-side (`src/auth/AuthScreens.tsx` and `LandingScreen`'s modal). `App.tsx` currently routes `Login` and `Register` to `AuthScreens`, but the primary path users see is through the Landing modal. Duplicate code drifts; pick one.

### C4. ЁЯЯб Storage-cleanup before login mutates state it doesn't own
`AuthContext.tsx:91-96`: `signIn` removes all `supabase`/`sb-` keys from localStorage before calling `signInWithPassword`. Supabase already manages its own storage; pre-wiping can invalidate PKCE verifiers mid-flow on slow networks.

### C5. ЁЯЯв `isLocalAuth` is effectively dead
`AuthContext.tsx:144`: `user?.email?.includes('local_')` тАФ nothing in the codebase creates an email with `local_` any more. The flag is always `false` but still exposed on the context.

## D. Supabase client (`src/lib/supabase.ts`)

### D1. ЁЯЯб `safeFetch` strips non-ISO-8859-1 chars from header values
The header-sanitizing fetch replaces `[^\x00-\xff]` with empty string. This works around a real bug in `@supabase/supabase-js` when a user has emoji in `user_metadata.name` (JWT header-encoded), but it silently corrupts auth tokens if a legitimate non-latin1 char ever ends up in a header. The regex range is also wrong тАФ `\x00-\xff` is a full single-byte range; ISO-8859-1 is `\x00-\xff`, so it's technically accurate but worth a comment explaining intent.

### D2. ЁЯЯб Client is created at module load time
`supabase.ts` throws at import time if env vars are missing. This makes dev setup friendlier (clear error) but breaks any tool that imports the module without an env (e.g. type-only imports in tests). Consider lazy init.

## E. Screens тАФ full list

### HomeScreen
- ЁЯЯа **Typo in icon name breaks Trophy menu item.** Line ~190: `name=" trophy-outline"` has a leading space тАФ Ionicons silently renders a "?" glyph on web, warning on native.

- ЁЯЯб `ControlButton` onPress is optional but "Fan" and "Bagasi" never pass one тАФ tapping them does nothing. If intended, add disabled styling.
- ЁЯЯб `bikeName` defaults to `user_metadata.motor`. After onboarding we write `motor: "${brand} ${model}"` to metadata, but there is no refresh when `bikes` table changes тАФ if user adds a bike in Garage, Home still shows the onboarded bike until app reload.
- ЁЯЯб Menu item mapping is confusing: "Kontrol & Mode Berkendara" navigates to `RideHistory`, "Keamanan & Sentry Mode" navigates to `Leaderboard`, "Servis & Perawatan" to `Garage`. None of these routes actually implement the labeled feature. These need either real destination screens or relabeled buttons.

### EventsScreen
- ЁЯФ┤ **Typo `borderExWidth` instead of `borderWidth` in two StyleSheets** (lines ~318 and ~331). TS2345. Runtime: no border renders, so category tabs and catItem pills have no outline.
- ЁЯЯа `submitGroup` is defined but **nothing calls it**. The UI exposes event creation but not group creation тАФ dead code or missing UI path.
- ЁЯЯа `joinGroup` uses `supabase.rpc('decrement', {})` with no args and no `id` field to decrement. This call will error silently (or succeed with wrong target); no error handling.
- ЁЯЯб `visibleEvents` filter on `tab !== 'SEMUA' && tab !== 'GRUP'` compares category as `tab.toLowerCase().replace(' ', '_')` (e.g. `"racing_meet"`) but the insert stores `eCat` with spaces ("Racing Meet"), and the OR branch checks against `tab.toUpperCase()`. Works only by accident.
- ЁЯЯб Group fetch doesn't filter `active` vs `pending` server-side; client filter trusts string equality.

### PartsScreen
- ЁЯЯб `partCard` width uses `(width - 48 - 12) / 2` computed from `Dimensions.get('window')` at module load. On resize (web) the card width doesn't update.
- ЁЯЯб Cart persistence keyed off item `id` only тАФ no quantity stored. `CartScreen` reconstructs qty in local state, so navigating back and forth loses quantity changes.
- ЁЯЯб "Buy" button falls back to WhatsApp with hardcoded number `6281234567890` тАФ not configurable.

### CommunityScreen
- ЁЯЯа **Duplicate `communities` variable declared**. Top of file line 45: `const communities: any[] = [];` (module-level), then inside the component line 60: `const [communities, setCommunities] = useState<any[]>([])`. The component's `useState` shadows the module-level array, so it works, but this is confusing code that should be cleaned up.
- ЁЯЯб `uploadToSupabase` is web-only (relies on `FileReader`, `canvas`, `toBlob`). No native path. Post button on native won't upload images.
- ЁЯЯб `handleLike` updates server-side `likes_count` via a non-atomic read-modify-write (`Math.max(0, likes_count + delta)` computed on client). Two simultaneous likes from different devices race тАФ last write wins.
- ЁЯЯб Fallback community list when DB is empty has hardcoded member counts (`"45.2K"`). These are cosmetic demo data; misleading in production.

### ProfileScreen
- ЁЯЯб `motorLabel` only reads primary bike. If user has bikes but none marked primary, it falls back to `motorFromMeta` which may be stale.
- ЁЯЯб `handleSignOut` is just `await signOut()` тАФ no confirmation prompt, no error feedback. One missed tap and user's logged out.
- ЁЯЯв `buildStats` runs on every render because it's not memoized. Cheap operation now, but will be heavy with hundreds of rides.

### LandingScreen
- ЁЯФ┤ **Typo `borderExWidth` in `featureCard` style** (line ~207). Same bug as EventsScreen. TS2345 error.
- ЁЯЯб `heroContainer` uses `Dimensions.get('window').height` set once at load. Doesn't respond to viewport resize on web.
- ЁЯЯб `doReg` shows `window.alert` for email verification only on web. Silent on native.

### GarageScreen
- ЁЯЯа **`handleAddBike` silently succeeds on DB error.** Lines ~91-103: catch block inserts a mock bike into local state with `Math.random().toString()` as id. User thinks bike is saved, but it's gone on refresh. Should surface the error.
- ЁЯЯа `setPrimaryBike` updates all bikes to `is_primary=false` including the target, then updates target to `true`. Two sequential writes with no atomicity тАФ if second fails, no primary bike exists.
- ЁЯЯб `fetchBikes` orders by `is_primary DESC` but no filter on `user_id` тАФ with RLS this is fine, without it every user sees every bike. Add `.eq('user_id', user.id)` explicitly.

### RideHistoryScreen
- ЁЯФ┤ **TS2339: `ts.card` doesn't exist.** Line 13: the inline `TeslaCard` wrapper references `ts.card` but the StyleSheet defines no `card` key (it has `rideCard`, `modalContent`, etc.). Works at runtime because the style is `undefined` (which RN tolerates), but the card renders without background/padding.
- ЁЯЯа `detectSegments` inserts segment_efforts without bounds checking тАФ an entry near both start and end of the same point produces negative times. The `elapsedSec = Math.max(1, (exitIdx - entryIdx) * 3)` caps at 1, but `avg_speed` is computed from `seg.distance_km / (1/3600)` = 3600 ├Ч distance, which stores absurd km/h values for very-short matches.
- ЁЯЯа `detectGroupRide` matches **every** other user's ride today regardless of overlap, creates a group_ride row unconditionally, then inserts all of them as members. So any ride posted after someone else's today becomes "group ride" with unrelated users.
- ЁЯЯа `simulateGPS` runs when permission is denied **or** when `coords.length === 0` after 6s. On slow networks or cold GPS fixes, real tracking gets replaced with fake data without notice.
- ЁЯЯб `snapToRoad` uses public OSRM demo server тАФ rate-limited, should not be used in production. 98-point cap may drop accuracy on long rides.
- ЁЯЯб `route_path` stored with `.slice(-500)` on save тАФ older points dropped, but `duration` and `distance` already reflect full ride. Relive animation will start mid-ride.
- ЁЯЯб Leaflet is loaded from `window.L` but never imported; relies on a script tag that is absent in `web/index.html`. Rendering fails on first mount unless Leaflet is loaded externally.

### RideReplayScreen
- ЁЯФ┤ **TS2322 on `canvas` `display: 'block'`**: React Native Web canvas is a raw DOM element but styled via RN StyleSheet тАФ TS warns that `display: 'block'` isn't a valid RN value (RN only supports `none | flex`). Runtime works on web, but typecheck fails and native builds will render nothing.

- ЁЯЯб Non-web fallback message is correct ("Relive is only available on web"), but the navigation badge still shows the menu item on native.
- ЁЯЯб No handling for rides with `route_path === null` тАФ `coords.length` check works but the UI shows an empty canvas with no explanation.

### LeaderboardScreen

- ЁЯЯб `group_rides` fetch joins `group_ride_members` but the inner select syntax assumes a foreign-key relationship Supabase infers automatically. Silently returns empty if the FK isn't set up.

### AchievementScreen

- ЁЯЯб `next` milestone picks the first non-unlocked achievement by array order, not by closest-to-unlock. Could show "Thousand KM Legend" when user is 1 km from "50KM Rider".

## F. Ride tracking / geolocation

### F1. ЁЯЯа Two haversine implementations
One in `RideHistoryScreen`, one referenced elsewhere. Consolidate in `src/lib/geo.ts`.

### F2. ЁЯЯа Distance filter is too aggressive
`processCoord`: increments only when `d >= 0.01 && d < 0.5`. The lower bound drops 10m-resolution GPS jitter on stationary тАФ good. The upper bound `< 0.5km` also silently drops any segment where the watchPosition callback happens after a 0.5km gap (e.g. brief tunnel). These gaps never get counted as distance.

### F3. ЁЯЯб Speed smoothing uses rolling mean of last 5 samples
Fine for steady riding, but on hard brake/accel the averaged display lags by ~15s.

## G. Types / navigation (`src/navigation/types.ts`)

### G1. ЁЯЯа All routes typed as `undefined` or optional
Every stack screen's params are `undefined` except `CommunityDetail` (which is also optional). Forces every `navigation.navigate('X', params)` to either trust `any` or cast. `RideReplay` actually receives a `ride` object (see B2) but its param type says `undefined`. This is the root cause of B2.

### G2. ЁЯЯв `MainTabParamList` used but never imported into bottom-tabs typing
`App.tsx` passes the generic to `createBottomTabNavigator<MainTabParamList>()` but individual screen components inside don't use `BottomTabScreenProps<MainTabParamList, 'Home'>` тАФ they all type navigation as `any`.

## H. SQL / schema

### H1. ЁЯФ┤ `schema.sql` has SQL typo that will fail to run
Line 42: `ALTER TABLE posts ENABLE ROWN LEVEL SECURITY;` тАФ `ROWN` should be `ROW`. Breaks the SQL file at the third ALTER. Users following README to bootstrap their DB get a partial schema.

### H2. ЁЯФ┤ `data.sql` has string escape bugs тАФ will fail to parse
Example line 70: `('MadeAdventure', 'Adv 150', 'Bromo-Semeru trip complete! Viewngakjub, nature poll ЁЯФе, 712, 92, 'тЫ░я╕П')` тАФ missing closing single quote after "nature poll ЁЯФе", extra comma pair, emoji left outside string. SQL editor throws syntax error before insert.

### H3. ЁЯЯа Schema missing tables actually used by the app
The app code reads/writes from: `profiles`, `bikes`, `rides`, `segments`, `segment_efforts`, `groups`, `group_members`, `group_rides`, `group_ride_members`, `group_posts`, `user_achievements`, `comments`. The schema file defines only 3 (events, parts, posts). Anyone running fresh from repo hits PostgREST "relation does not exist" on almost every screen.

### H4. ЁЯЯа No foreign keys between `rides`, `segment_efforts`, `group_ride_members`, `bikes`
Reference the `users` table but no FK declared, so orphan rows after user deletion.

### H5. ЁЯЯб `.affiliate_url` / `.link` / `.condition` / `.sold` / `.rating` columns read by client are not in the `parts` schema
`PartsScreen` reads `.affiliate_url`, `.badge`, `.sold`, `.rating`, `.image_url`, `.seller_name`, but schema only has `title, description, price, condition, category, seller_name, location, status, image_emoji`. Either seed data adds more columns at runtime (ALTER TABLE not in file) or these render as `undefined`.
